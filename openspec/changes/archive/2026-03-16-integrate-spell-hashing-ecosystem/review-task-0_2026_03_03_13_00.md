## Unit B — Normalization & Hash Pipeline

*Scope: canonical_spell.rs 303–322, 333–400, 528–545, 690–866, 869–1006*

### Pass 1 — Spec Compliance & Correctness

- No issues. normalize() calls migrate_to_v2() first. prune_metadata_recursive strips the fields required by the canonical-serialization contract. Bulk migration path (normalize then to_canonical_json_pre_normalized) produces v2 canonical hash without double normalization.

### Pass 2 — Robustness & Edge Cases

- **Minor** **[FIXED]** — prune_metadata_recursive has no recursion-depth limit. Safe today because input is always to_value(CanonicalSpell).  
  **Line:** 334–400. **Recommendation:** If ever used on untrusted or arbitrary Value, add a max-depth parameter or iterative implementation. *Doc comment added: no depth limit, only used on to_value(CanonicalSpell); note to add max-depth if used on untrusted JSON.*

*Other checks:* compute_hash() on un-normalized/v1 spell produces correct v2 hash. Double normalization is safe; idempotency holds for already-v2 spells. normalize_string and match_schema_case handle empty string and Unicode safely.

### Pass 3 — Code Quality & Maintainability

- **Important** **[ADDRESSED — doc]** — default_schema_version() returns CURRENT_SCHEMA_VERSION (2). When canonical_data JSON lacks schema_version, the field deserializes to 2 and migrate_to_v2() is skipped. Legacy or external v1 blobs that omit the key are never migrated.  
  **Line:** 250–261, 692–694. **Recommendation:** Either use default_schema_version() = MIN_SUPPORTED_SCHEMA_VERSION (1) so missing key goes through migration, or document that external/manual JSON must include schema_version for correct migration. *Doc comment now explains consequence and that only hand-crafted/import JSON is affected.*

- **Minor** **[FIXED]** — camel_to_snake: consecutive capitals are split per character (e.g. "XMLParser" → "x_m_l_parser"). Safe for current schema keys. Optional: add a short comment and a test to lock behavior.  
  **Line:** 420–430. *Doc comment and test_camel_to_snake added.*

- **Minor** **[OPEN — optional]** — normalize() is ~177 lines; readability could improve by extracting helpers (e.g. normalize_primitive_and_spec_fields, apply_lean_hashing_pruning). Word-boundary check is duplicated in replace_word_boundary_alias and find_case_insensitive_alias_match; could be shared.  
  **Line:** 690–866, 456–526.

## Unit C — Batch Migration & Tauri Command

*Scope: canonical_spell.rs 4230–4371, lib.rs*

### Pass 1 — Spec Compliance & Correctness

- **Important** **[FIXED]** — The batch only migrates spells with schema_version < 2. Spells that are already v2 but have content_hash NULL are skipped (result.skipped). tasks.md 0.1.3 and design.md require that all spell.content_hash values be v2 compliant.  
  **Line:** 4257 (condition if spell.schema_version < 2), 4311–4312 (skip branch). **Recommendation:** Either (a) treat rows as needing migration when schema_version < 2 OR content_hash IS NULL and run the same normalize/validate/hash/update path, or (b) document that this command is "v1→v2 only" and that another path (e.g. --recompute-hashes) is responsible for v2 rows with NULL hash, and ensure that path is part of Task 0. *Condition is now `schema_version < 2 || hash_db.is_none()`; v2 rows with NULL hash are backfilled.*

*Other checks:* DB columns updated correctly (canonical_data, content_hash, schema_version, updated_at). Progress emitted; Tauri command registered and documented as migration trigger.

### Pass 2 — Robustness & Edge Cases

- **Important** **[FIXED]** — Same as Pass 1: v2 spells with NULL content_hash are never backfilled.  
  **Line:** 4310–4312.

- **Minor** **[FIXED]** — PROGRESS_BATCH_SIZE = 10 yields 1000 window events for 10k spells. May stress UI or event pipeline.  
  **Line:** 4229, 4324–4326. **Recommendation:** Consider 50–100 or make configurable; document that progress is "every N spells" so frontend can throttle if needed. *Set to 50; doc added.*

*Other checks:* On UNIQUE constraint violation the transaction rolls back; no partial updates. Mid-batch UPDATE failure causes Err and rollback. All-or-nothing behavior is correct.

### Pass 3 — Code Quality & Maintainability

- **Important** **[FIXED]** — The notes character limit 2048 is a magic number in six places (truncation, assertions, error messages).  
  **Line:** 644, 648, 309, 535, 4264, 3643. **Recommendation:** Add a named constant, e.g. SAVING_THROW_NOTES_MAX_CHARS = 2048, and use it everywhere. *Constant `SAVING_THROW_NOTES_MAX_CHARS` added and used everywhere.*

- **Minor** **[FIXED]** — run_migration_batch_impl is ~135 lines with nested match; could be split into collect_migration_updates and apply_migration_updates for readability.  
  **Line:** 4234–4391. *Refactored into collect_migration_updates and apply_migration_updates (MigrationUpdate struct); same transaction/rollback and progress semantics.*

- **Minor** **[FIXED]** — MigrationFailure error messages: "JSON error" and "Hash error" are implementation-oriented. Prefer user-oriented wording (e.g. "Could not serialize spell after migration") while keeping the underlying error for debugging.  
  **Line:** 4289, 4298.

- **Minor** **[FIXED]** — Notes truncation message in batch impl should match other occurrences ("Saving throw notes truncated during migration (exceeded 2048 characters)").  
  **Line:** 4264 vs 309, 535. *Batch now uses same message and SAVING_THROW_NOTES_MAX_CHARS.*

*pub(crate)* visibility and PROGRESS_BATCH_SIZE as a named constant are appropriate.

## Unit D — Tests

*Scope: canonical_spell.rs lines 3577–4227*

### Pass 1 — Spec Compliance & Correctness

- **Important** **[FIXED]** — Task 0.1.3 requires hashes recalculated "including raw_legacy_value". No test asserts that stored content_hash after migration equals recomputed v2 hash for a spell that has raw_legacy_value (e.g. 5e casting time). test_migration_batch_content_hash_is_v2_canonical uses a spell without casting_time; test_migration_batch_5e_unit_remapping_in_db checks raw_legacy_value in canonical_data but does not assert content_hash == spell.compute_hash().  
  **Line:** 4019–4069, 4075–4142. **Recommendation:** In test_migration_batch_5e_unit_remapping_in_db, after asserting casting_time and raw_legacy_value, add: load content_hash from DB and assert content_hash == spell.compute_hash().expect("recompute"). Alternatively add a dedicated test that migrates a 5e-unit spell and asserts stored hash equals recompute. *test_migration_batch_5e_unit_remapping_in_db now asserts content_hash == spell.compute_hash().*

*Other checks:* 0.1.1 trigger is documented; batch logic is tested via run_migration_batch_impl. 0.1.4 (5e remapping, dm_guidance cleanup) covered by dedicated tests. Test schema matches columns the migration reads and writes.

### Pass 2 — Robustness & Edge Cases

- **Important** **[FIXED]** — No test covers the case where a spell row has schema_version = 2 and content_hash IS NULL. The implementation skips such rows; they never get content_hash set.  
  **Line:** 3577–4227, impl 4258, 4310–4313. **Recommendation:** Add a test that inserts a row with schema_version = 2 and content_hash NULL, runs run_migration_batch_impl, and asserts either (a) row is skipped and still has NULL content_hash, or (b) if product goal is to backfill, that such rows are backfilled and the test reflects that. *test_migration_batch_v2_null_hash_is_backfilled added; asserts backfill and content_hash == recomputed.*

- **Minor** **[FIXED]** — Progress callback: only truncation-failure path is tested. Success and deserialization-failure paths use the same callback site; optional: add a short test that runs a successful batch and asserts at least one progress event, or add a comment that the single progress test is representative.  
  **Line:** 4325–4327, 3963–4012. *test_migration_batch_progress_emitted_on_success added.*

*Other checks:* test_migration_batch_db_failure_rollback is complete (Err, no commit, COUNT content_hash IS NOT NULL = 0). No flaky or environment-dependent tests.

### Pass 3 — Code Quality & Maintainability

- **Important** **[FIXED]** — The same CREATE TABLE spell schema is repeated in six tests.  
  **Line:** 3844–3855, 3909–3921, 3969–3980, 4024–4035, 4081–4092, 4156–4167. **Recommendation:** Add a shared helper, e.g. setup_migration_db(with_unique_hash: bool) -> Result<Connection, rusqlite::Error>, and use it in all six tests. *setup_migration_db() and setup_migration_db_with_unique_hash() added; all batch tests use them.*

- **Important** **[PARTIAL]** — Several tests have spec/task info only in doc comments, not in the test name (e.g. Task 0.1.4, dm_guidance, Priority C/D).  
  **Line:** 3576–3840, 3906–3967. **Recommendation:** Include a short spec id in the name where applicable, e.g. test_migration_batch_task_0_2a_content_hash_is_v2_canonical, test_migration_batch_task_0_2c_dm_guidance_cleanup_in_db. *Test names remain descriptive; Task/Priority/spec id are in doc comments.*

- **Important** **[FIXED]** — Doc comments are inconsistent: some tests have Task/Priority, others have none.  
  **Line:** 3576, 3616, 3646, 3682, 3706, 3731, 3740, 3837, 3906, 3964, 3909–3955. **Recommendation:** Add a single-line doc comment to every migration-related test (e.g. "Task 0.1.x" or "Priority C/D" or "Task 0 Unit 2(a/b/c)"). *All migration-related tests now have Task/Priority or Task 0 Unit doc comments.*

- **Minor** **[FIXED]** — Optional: add assertion helpers (e.g. assert_migrated_row_has_hash) if tests grow. Optional: add (TG…) to all Priority/Task docs for consistency.
