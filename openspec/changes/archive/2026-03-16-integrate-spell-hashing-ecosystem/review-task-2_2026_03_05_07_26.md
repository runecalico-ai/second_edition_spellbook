## Pass 1: Architecture & Compliance

### 1.1 Import Pipeline Alignment
The implementation in `import.rs` (Task 2.1) follows the prescribed pipeline order:
- **Parse/Classify**: `parse_and_classify_payload` correctly distinguishes between single spells and bundles using the top-level `spells` array.
- **Normalization/Truncation**: `normalize_truncate_metadata` is called early in the preview phase, enforcing the limits of 100 tags and 50 source references before hashing or validation.
- **Schema Validation**: Per-spell `schema_version` checks are performed in `process_spell`, providing a warning for forward-compatibility without failing the import.
- **Deduplication Logic**:
    - **Hash-First Identity**: The code correctly lookups by `content_hash` before checking by name.
    - **Merge Logic**: Deduplication (skip-insert) triggers `merge_tags` and `merge_canonical_data_source_refs`.
    - **Intra-bundle Deduplication**: `seen_hash_in_batch` ensures that if multiple spells with the same hash appear in one bundle, only the first is inserted (and others are merged).

### 1.2 Export Correctness
The implementation in `export.rs` (Task 2.2) adheres to the export specifications:
- **Single-Spell**: `export_spell_as_json` emits a flat `CanonicalSpell` without the bundle envelope. It correctly sets the `id` to the `content_hash` and includes `schema_version`.
- **Bundle**: `export_spell_bundle_json` uses the `SpellBundleExport` envelope, including `schema_version` and `bundle_format_version`.
- **Integrity**: Both paths correctly reject exports if the `content_hash` or `canonical_data` is missing (prompting a migration).

### 1.3 Conflict Resolution Behavior
- **Numeric Suffixes**: `find_unique_name_for_keep_both` implements the "(1)", "(2)" suffixing logic by checking both the DB and the current batch.
- **Replace with New**: `replace_with_new_impl` performs the UPDATE and correctly executes the cascade to `character_class_spell` and `artifact` tables as required by Task 3.1.67.

## Pass 2: Correctness & Logic

### 2.1 Metadata Merging (Edge Cases)
- **Tag Merging**: `merge_tags` correctly handles cases with `None` or empty strings. It uses a `Vec` for sorting and `dedup()`, ensuring the final string is clean.
- **SourceRef Merging**: `merge_canonical_data_source_refs` uses a `HashSet` with `source_ref_dedup_key` for O(1) deduplication during the merge. It correctly prioritizes existing references and caps the result at 50.

### 2.2 URL Sanitization & Security
- **Allowlist**: `validate_source_ref_url` strictly enforces `http`, `https`, and `mailto`.
- **Sanitization**: `sanitize_url_for_display` uses regex to strip HTML tags and collapse whitespace, mitigating potential XSS/injection in displayed URLs.
- **Policy Enforcement**: `process_source_ref_urls` correctly implements both `DropRef` and `RejectSpell` policies, with appropriate warnings added to the result.

### 2.3 SQL & Transactional Integrity
- **Transactional Safety**: `apply_import_spell_json_impl` uses an `unchecked_transaction()`. A critical fix was noted in verification: replace failures now propagate error results (`?`), ensuring the transaction is rolled back and no partial updates (e.g., half-cascaded hashes) are committed.
- **Parameterized Queries**: All queries (including FTS and metadata lookups) use `params!` or similar, preventing SQL injection.
- **Replace Hash Collision**: `replace_with_new_impl` prevents a hash collision where "Replace with New" would cause two rows to have the same hash but potentially different names (which should have been a merge).

### 2.4 Error Handling
- **Graceful Failure**: The two-phase import (Preview -> Apply) allows the system to identify malformed spells/bundles before any DB writes occur.
- **Tamper Detection**: Comparing the imported `id` with the recomputed `content_hash` correctly identifies tampering while preferring the recomputed hash.

## Pass 3: Quality & Maintainability

### 3.1 Code Structure & Readability
- **Modularity**: Logic is well-factored into small, focused functions (`source_ref_dedup_key`, `normalize_truncate_metadata`, `process_spell`).
- **Dryness**: `canonical_spell_to_flat_row` and `diff_canonical_vs_detail` centralize the mapping between the nested `CanonicalSpell` and the flat DB schema.
- **State Management**: Using `spawn_blocking` for DB-heavy paths in `import.rs` and `export.rs` is correct for Tauri's async requirements.

### 3.2 Documentation
- **Inline Comments**: The file contains clear comments mapping functions to specific Tasks (e.g., "// Task 2.1: parse, classify, normalize, hash").
- **Variable Naming**: Names like `seen_hash_in_batch` and `resolutions` accurately describe their role.

### 3.3 Potential Improvements (Nitpicks)
- **Regex Performance**: `sanitize_url_for_display` recompiles the regex on every call. For large imports, this might be slightly optimized by using `lazy_static!` or `OnceLock`, although current performance is likely acceptable given the context.
- **Metadata Merge Limit Logic**: `merge_canonical_data_source_refs` returns early if the limit is reached during existing ref processing. This is correct as per "existing first" policy.
