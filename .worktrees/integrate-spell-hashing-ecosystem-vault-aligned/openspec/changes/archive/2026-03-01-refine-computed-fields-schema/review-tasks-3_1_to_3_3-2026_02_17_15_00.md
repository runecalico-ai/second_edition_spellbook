# Code Review: tasks 3.1-3.3 (refine-computed-fields-schema)

## Overview
A three-pass, in-depth code review was conducted on the implementation of tasks 3.1, 3.2, and 3.3 spanning `src/types/spell.ts`, `DamageForm.tsx`, and `SavingThrowInput.tsx`.

While much of the frontend mapping and `notes`/`sourceText` component integration is sound, there are **two blocking defects** violating the spec's strict data semantic rules that must be resolved prior to proceeding with further implementation.

> *Display `rawLegacyValue` as a read-only labelled annotation when populated, for all `kind` values... For `dm_adjudicated` specifically: no SingleSave sub-form is shown; `notes` is the sole editable narrative field.*

---

## Pass 2: Logical & Semantic Check

1. **Safety of `rawLegacyValue` Mutations**: The decision to render an editable `<input>` for `rawLegacyValue` under `dm_adjudicated` presents a data integrity threat. `rawLegacyValue` represents the unmodified, un-normalized string retrieved dynamically at parse/migration time. Giving users the ability to overwrite it directly breaks the implicit contract of it being the "immutable source text". This field must be completely read-only. Editing behavior must flow strictly into the `notes` field.

2. **Casting Time Mappings Check**: The retention of the 5E units in the `castingTimeToText()` display logic isn't intrinsically harmful if they can never be created, but keeping them in the `CASTING_TIME_UNIT_LABELS` dictionary means any connected `<select>` field iterating through `Object.entries(CASTING_TIME_UNIT_LABELS)` will mistakenly include them for users to click. They must be aggressively purged.

---

## Pass 3: Impact & E2E Verification

If these branch issues were merged as-is, E2E tests would fail on two fronts:
1. Playwright tests expecting the dropdown `Casting Time` component to be empty of `"action"` / `"bonus_action"` / `"reaction"` options would find them readily selectable.
2. The user experience invariants tests for `SavingThrowInput.tsx` would complain about a writable text input field meant for legacy data display appearing upon checking "DM Adjudicated".

Additionally, we risk corrupting `canonical_data` hashes (which include `rawLegacyValue`) by making `rawLegacyValue` editable.

## Required Action Items (Fixes to Apply)

**Fix 1: In `src/types/spell.ts`**
Remove `"action" | "bonus_action" | "reaction"` from:
- The `CastingTimeUnit` union type block.
- The `CASTING_TIME_UNIT_LABELS` record map.
- The `unitLabels` mapping inside the `castingTimeToText()` function.

**Fix 2: In `src/ui/components/structured/SavingThrowInput.tsx`**
Replace the entire conditional `spec.kind === "dm_adjudicated"` ternary wrapper with an unconditional read-only view.

Replace lines 140-157:
```tsx
      {spec.kind === "dm_adjudicated" ? (
        <input ... />
      ) : (
        spec.rawLegacyValue && (
          <div ...>...</div>
        )
      )}
```

With exactly:
```tsx
      {spec.rawLegacyValue && (
        <div className="flex items-center gap-2 px-2 py-1 bg-amber-900/10 border border-amber-900/30 rounded text-[10px] text-amber-200/70 italic">
          <span className="font-bold uppercase not-italic">Original source text:</span>
          <span>{spec.rawLegacyValue}</span>
        </div>
      )}
```
