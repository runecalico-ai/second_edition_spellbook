## Unit A — FTS5 Migration SQL

*Scope: `db/migrations/0014_fts_extend_canonical.sql`*

### Pass 1 — Spec Compliance & Correctness

- No issues. All Task 1.1 checklist items are satisfied:
  - `DROP TRIGGER IF EXISTS` for all three triggers. ✓
  - `DROP TABLE IF EXISTS spell_fts`. ✓
  - Virtual table declares all 14 spec-required columns in the correct order with `content='spell'` and `content_rowid='id'`. ✓
  - Repopulation uses explicit `INSERT INTO ... SELECT ... json_extract(...) FROM spell` (not `VALUES('rebuild')`). ✓
  - `spell_ad` (AFTER DELETE) trigger passes `old.*` values — including `json_extract(old.canonical_data, ...)` — in the FTS5 `'delete'` command, not empty strings. ✓
  - `spell_au` (AFTER UPDATE) trigger is structured as delete-old then insert-new; both sections use `old.*` / `new.*` correctly. ✓
  - All 8 canonical text columns use the correct JSON paths per spec. ✓

### Pass 2 — Robustness & Edge Cases

- **Minor** — Concatenated canonical fields produce a spurious single space when all sub-fields are absent or `canonical_data` is NULL.  
  **Pattern:** `COALESCE(..., '') || ' ' || COALESCE(..., '')` → `' '` when both are NULL.  
  **Impact:** Functionally harmless — FTS5 tokenizer reduces a single space to zero tokens; no search mismatch results. Any future `SELECT canonical_casting_time_text FROM spell_fts WHERE rowid = ?` returns `' '` instead of `''` or NULL. No fix required; acceptable for FTS indexing.

- **Minor** — `content='spell'` + derived columns: FTS5 auxiliary functions `highlight()` and `snippet()` will silently fail for canonical_* columns (indices 6–13) because those columns do not exist on the `spell` content table. The existing comment warns against `SELECT * FROM spell_fts` but does not mention this.  
  **Line:** 29–33. **Recommendation:** Extend the warning comment:

  ```sql
  -- NOTE: highlight() and snippet() auxiliary functions will also fail for
  -- canonical_* column indices (6–13) since those columns do not exist on
  -- the spell table. Use MATCH for search; join back to spell for display.
  ```

### Pass 3 — Code Quality & Maintainability

- **Minor** — `canonical_range_text`, `canonical_duration_text`, and `canonical_area_text` use bare `json_extract(...)` (potentially NULL), while the other five canonical columns use COALESCE-concatenation (always a non-null string). The inline comments `-- single value; NULL when absent` are present in the repopulation SELECT but absent from the trigger VALUES sections. A maintainer adding a new field might not understand the asymmetry.  
  **Recommendation:** The top-level maintenance comment is sufficient; optionally add a one-line note in the trigger block explaining that single-source columns store NULL while multi-source columns are always strings.

## Unit B — FTS5 Query Builder + Search Command

*Scope: `apps/desktop/src-tauri/src/commands/search.rs` lines 1–304*

### Pass 1 — Spec Compliance & Correctness

- **Important** — Leading `NOT` in a query activates advanced mode and emits `NOT "ice"` (e.g. for query `"NOT ice"`), which is **invalid FTS5 syntax**. FTS5's `NOT` is a binary infix operator — `<query> NOT <query>` — not a unary prefix. A leading `NOT` with no left operand will produce a runtime SQLite error instead of falling back to basic mode as the spec requires.  
  **Lines:** 59–63 (`try_build_advanced_fts_query`); test at line 722 documents the incorrect expectation ("Leading NOT (negation) is valid in FTS5").  
  **Fix:** Broaden the first-token guard to reject all FTS5 operators, not just binary ones:

  ```rust
  // FTS5 NOT is also infix-only; reject any leading operator.
  if is_fts_operator(tokens[0]) {
      return None;
  }
  ```

  Also update the doc comment and `test_advanced_mode_not_operator` to expect the basic-mode fallback (`"\"NOT ice\""` instead of `"NOT \"ice\""`).

- No other spec compliance issues. MATCH always bound as a single `?` parameter. `bm25()` ordering is correct (lower = more relevant, `ASC`). JOIN with `spell_fts` only when a text query is present. All filter values are bound parameters. ✓

- **Note on `escape_fts_phrase_content`:** The function only doubles `"` and relies on FTS5's quoted-phrase semantics to protect `*`, `(`, `)`, `^`, `:`, `-`, `+`. This is functionally correct — inside a quoted string FTS5 passes all non-`"` characters to the tokenizer where they have no query-level meaning. The doc comment ("Only `"` needs to be doubled") diverges from the spec's phrasing ("escape ALL FTS5 special characters") without explaining the relationship; see Pass 3.

### Pass 2 — Robustness & Edge Cases

- **Important** — LIKE wildcard characters `%` and `_` in filter values are not escaped before binding.  
  **Affected lines:** All six `LIKE ?` filter branches for `schools`, `spheres`, `class_list`, `source`, `components`, `tags` (lines ~195–260).  
  **Impact:** The values are safe from SQL injection (bound parameters), but user input containing `_` is interpreted as "any single character" and `%` as "any sequence," broadening matches unexpectedly. For school/sphere dropdowns this is low risk; for free-form text filters (`source`, `tags`) it is a real usability and correctness concern.  
  **Fix:** Escape LIKE wildcards before formatting and declare an escape character:

  ```rust
  let escaped = value.replace('\\', "\\\\").replace('%', "\\%").replace('_', "\\_");
  sql.push_str(&format!("{}source LIKE ? ESCAPE '\\\\'", col));
  params.push(Box::new(format!("%{}%", escaped)));
  ```

  Apply consistently across all six LIKE filter branches.

- **Minor** — LIMIT 100 silently truncates filter-only result sets ordered alphabetically by name. For a library with hundreds of spells across a given school, a "show all Evocation spells" filter could silently drop the tail. The BM25-ranked path has a clear justification for capping; the filter-only path does not. See Pass 3, P3-3.

- All other edge cases pass: empty/whitespace query is correctly handled by the `has_text_query` guard before `build_fts_query` is called. NEAR is always treated as a content token. No user text reaches SQL outside the bound-parameter system. ✓

### Pass 3 — Code Quality & Maintainability

- **Minor** — `escape_fts_phrase_content` doc comment says "Only `"` needs to be doubled" without explaining how the phrase-wrapping covers the other spec-required special characters. A future maintainer reading the spec alongside this comment will see an apparent contradiction.  
  **Line:** 17–18. **Recommendation:** Add a bridging sentence explaining that phrase wrapping satisfies the spec's escaping goal for all non-`"` characters, because they have no query-level meaning inside FTS5 quoted strings.

- **Minor** — The `col` prefix approach (`"s."` or `""`) must be manually prepended to every column reference in every filter branch. A new filter added without `col` will compile silently and only fail at runtime with an "ambiguous column name" SQLite error, and only when both a text query and that specific filter are active.  
  **Lines:** ~163–260. **Recommendation:** Add a prominent `// IMPORTANT: All filter column references must use the 'col' prefix when joining spell_fts.` comment directly above the filter block to make the invariant explicit.

- **Minor** — LIMIT 100 appears twice as a magic number with different semantic justifications (relevance cap vs. alphabetical completeness cap).  
  **Lines:** 268–270. **Recommendation:** Extract as a named constant:

  ```rust
  const SEARCH_RESULT_LIMIT: usize = 100;
  ```

- **Minor** — `collect_facet_entries` uses `HashSet` + explicit `.sort()` where `BTreeSet` would maintain sorted order on insertion.  
  **Lines:** ~129–148. **Recommendation:** Switch to `BTreeSet<String>` to eliminate the explicit sort and remove the `mut` on the `sorted` binding. Minor quality improvement; no functional impact.

## Unit C — Tests

*Scope: `apps/desktop/src-tauri/src/commands/search.rs` lines 486–989*

### Pass 1 — Spec Compliance & Verification Plan Coverage

- **Important** — Verification plan test 2 is not covered: *"Given spells 'Fireball' and 'Frostbolt', when searching for 'Fire', result must include 'Fireball' and NOT include 'Frostbolt'."* No test exercises basic single-token FTS matching independently of phrase logic or filters. A regression in tokenization or index configuration would go undetected.  
  **Fix:** Add a dedicated test:

  ```rust
  #[test]
  fn test_search_single_token_matches_relevant_spell() {
      let conn = setup_search_db();
      insert_spell(&conn, 1, "Fireball", "A blazing orb of fire");
      insert_spell(&conn, 2, "Frostbolt", "A shard of ice and frost");
      let ids = search_ids(&conn, "fire");
      assert!(ids.contains(&1), "'Fireball' must match single-token query 'fire'");
      assert!(!ids.contains(&2), "'Frostbolt' must NOT match single-token query 'fire'");
  }
  ```

- **Minor** — Only `canonical_range_text` is exercised across all trigger lifecycle tests (insert, update, delete). The migration indexes 8 canonical fields; if any of the remaining 7 JSON paths (`$.duration.text`, `$.damage.source_text`, etc.) is misconfigured, no test would catch it. The verification plan does not mandate coverage of each field individually, so this is a coverage gap rather than a spec non-compliance.

- Remaining verification plan items are covered: tests 1, 3, 4, 5, 6, 7. ✓

### Pass 2 — Robustness & Edge Cases

- **Important** — `build_fts_query("")` (empty string) is not tested anywhere. The function returns `String::new()` for an empty input, but its behavior when passed to `search_keyword_with_conn` (which guards with `has_text_query`) is not verified end-to-end. Users clearing the search box can trigger this path.  
  **Fix:** Add:

  ```rust
  #[test]
  fn test_empty_query_returns_empty_string() {
      assert_eq!(build_fts_query(""), "");
  }

  #[test]
  fn test_search_empty_query_returns_results_without_fts() {
      let conn = setup_search_db();
      insert_spell(&conn, 1, "Fireball", "A blazing orb of fire");
      let ids = search_ids(&conn, "");
      assert!(ids.contains(&1), "empty query should return spells via non-FTS path");
  }
  ```

- **Important** — `test_malformed_not_or_falls_back_to_basic` and `test_malformed_not_or_after_term_falls_back_to_basic` have identical bodies:

  ```rust
  assert_eq!(build_fts_query("fire NOT OR ice"), "\"fire NOT OR ice\"");
  ```

  The second test's name implies a distinct scenario (e.g., `"fire OR NOT ice"`) was intended but was not implemented, leaving a coverage gap and misleading future maintainers.  
  **Lines:** ~769 and ~780. **Fix:** Replace the duplicate with the intended distinct case:

  ```rust
  #[test]
  fn test_malformed_or_not_after_term_falls_back_to_basic() {
      assert_eq!(build_fts_query("fire OR NOT ice"), "\"fire OR NOT ice\"");
  }
  ```

- **Minor** — No test for multi-word tokens in advanced mode (e.g., `"fire wizard AND NOT ice demon"`). All advanced-mode tests use single-word terms. If whitespace-splitting or quoting has a boundary condition, it goes undetected.

- **Minor** — No test for filter-only search (empty text query with populated filters). `search_keyword_with_conn` takes the non-FTS path when query is empty; this code path has no dedicated test.

- **Minor** — `test_search_near_always_basic_mode` asserts `ids.is_empty()` because neither inserted spell contains the consecutive phrase "fire NEAR shield". This is correct but weakly conclusive: the test cannot distinguish between "NEAR was escaped to a literal word" and "NEAR acted as an FTS5 operator and found nothing anyway." A spell containing those three words sequentially in its description would make the assertion more diagnostic.

### Pass 3 — Code Quality & Maintainability

- **Minor** — `setup_fts_db()` and `setup_search_db()` are structurally duplicated: both open an in-memory connection, define overlapping `spell` table schemas (the latter a superset), and load the same migration SQL via `include_str!`. The migration path literal appears twice. Moving or renaming the migration file requires two updates.  
  **Recommendation:** Unify by making `setup_fts_db()` delegate to `setup_search_db()` (the full schema is a superset and trigger tests only query `spell_fts`), or extract the migration-load step:

  ```rust
  fn setup_search_db() -> Connection { /* full schema + migration */ }
  fn setup_fts_db() -> Connection { setup_search_db() }
  ```

- **Minor** — No test verifies BM25 relevance ordering: a spell whose name matches the query should rank above one that only matches in description. Relevance ordering is an observable contract and could silently regress.

- **Minor** — `test_fts_null_canonical_data` doc comment mentions only description indexing but also asserts name indexing. The second assertion's intent should be stated explicitly in the comment.
